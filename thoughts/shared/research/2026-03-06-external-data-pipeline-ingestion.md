# Research: External Data Pipeline Ingestion for Octobox

**Date**: 2026-03-06T14:10:00+01:00
**Researcher**: Claude
**Git Commit**: 801e64f1d
**Branch**: hotfix/email-cache
**Repository**: open-webui

## Research Question

A customer (Octobox, serving government organizations) has their own document processing pipeline that produces clean parsed text with metadata. How can we let them push/pull this pre-processed data into our Weaviate vector DB for RAG retrieval? What are common approaches, and what would work best given our architecture?

## Summary

Our codebase already has the building blocks for external data ingestion:

1. **Existing external pipeline integration** (`external_retrieval.py`) — currently a "pull" model where Open WebUI sends text to an external `/chunk` endpoint. This could be extended.
2. **`save_docs_to_vector_db()`** — the core function that handles embedding + vector storage, with a `split=False` option that skips chunking (perfect for pre-chunked data).
3. **Typed knowledge bases** — the `type` column on the Knowledge model already supports custom types (currently "local" and "onedrive").
4. **`/process/text`** endpoint — accepts raw text and embeds it. Already close to what's needed.
5. **`/process/files/batch`** endpoint — batch processing that accepts `FileModel` objects.

The recommended approach is a **push API** — a new endpoint on Open WebUI that Octobox calls to push pre-processed documents. This is simpler, more reliable, and gives Octobox control over when and what they push.

## Detailed Findings

### Current Architecture

#### Ingestion Flow

```
File Upload → Loader (text extraction) → Chunking → Embedding → Weaviate Insert
```

The key function is `save_docs_to_vector_db()` at `backend/open_webui/routers/retrieval.py:1352`:

- Accepts `Document` objects (langchain_core)
- `split=True/False` — controls whether to chunk (False = pre-chunked data)
- `add=True/False` — controls whether to add to existing collection
- Handles dedup via hash-based checks
- Generates embeddings via configured embedding function
- Inserts into VECTOR_DB_CLIENT (Weaviate in our case)

#### Weaviate Schema

Collections are created with these properties (`backend/open_webui/retrieval/vector/dbs/weaviate.py:98-137`):

- `text` (TEXT) — the chunk content
- `file_id` (TEXT) — reference to file record
- `name` (TEXT) — filename
- `source` (TEXT) — source identifier
- `created_by` (TEXT) — user ID
- `moddate`, `creationdate` (TEXT) — document dates
- `onedrive_item_id`, `onedrive_drive_id` (TEXT) — OneDrive metadata

Vectors are self-provided (not auto-generated by Weaviate).

#### Existing External Pipeline

`backend/open_webui/routers/external_retrieval.py` — sends extracted text TO an external pipeline for chunking, then embeds the chunks locally. This is a "fan-out for chunking" pattern, not an ingestion API.

### Approach Options

#### Option A: Push API (Recommended)

Add a new endpoint to the retrieval router that accepts pre-processed documents:

```
POST /api/v1/retrieval/ingest
Authorization: Bearer <api_key>

{
  "collection_name": "octobox_documents",   // or knowledge_id
  "documents": [
    {
      "text": "chunk content here...",
      "metadata": {
        "source": "policy-doc-123.pdf",
        "title": "Privacy Policy 2026",
        "page": 3,
        "octobox_doc_id": "ext-abc-123"
      }
    },
    ...
  ],
  "skip_chunking": true    // data is already chunked
}
```

**Why this is best:**

- Octobox controls the push timing and content
- No polling, no webhook complexity
- Reuses `save_docs_to_vector_db()` with `split=False`
- Fits naturally into the existing FastAPI router pattern
- API key auth aligns with government security requirements
- Idempotent via hash-based dedup (already built in)

**Implementation steps:**

1. Add new knowledge base type `"external"` or `"octobox"`
2. Create `POST /api/v1/retrieval/ingest` endpoint
3. Accept pre-chunked documents with metadata
4. Convert to `Document` objects, call `save_docs_to_vector_db(split=False)`
5. Create/update File records to maintain the KB ↔ File relationship
6. Return ingestion status with per-document results

#### Option B: Pull API (Polling Octobox's API)

Open WebUI periodically polls Octobox's API for new documents:

```
GET https://octobox-api.example.com/documents?since=<timestamp>
→ Returns list of pre-processed document chunks
```

**When this makes sense:**

- If Octobox prefers not to call our API
- If documents are continuously streaming
- Similar to our OneDrive sync pattern (periodic worker)

**Downsides:**

- Requires a background worker (like OneDrive sync)
- More complex state management (cursor, last-sync timestamps)
- Octobox needs to maintain a stable API
- Polling interval creates latency

#### Option C: Webhook + Push Hybrid

Octobox sends a webhook notification, Open WebUI then pulls the data:

```
POST /api/v1/retrieval/webhook/octobox
→ {"event": "documents_ready", "batch_id": "xyz"}

Then Open WebUI calls:
GET https://octobox-api.example.com/documents/batch/xyz
```

**Too complex for the use case.** Only makes sense if documents are very large and need streaming.

### Recommended Architecture

```
┌──────────────────┐          ┌──────────────────────────┐
│   Octobox Data   │          │      Open WebUI          │
│   Pipeline       │          │                          │
│                  │          │  POST /api/v1/retrieval/  │
│  Parse → Clean   │ ──push──►│       ingest             │
│  → Chunk         │          │         │                │
│                  │          │    ┌────▼────┐           │
└──────────────────┘          │    │ Validate │           │
                              │    │ + Auth   │           │
                              │    └────┬────┘           │
                              │    ┌────▼────────┐       │
                              │    │ Embed        │       │
                              │    │ (our model)  │       │
                              │    └────┬────────┘       │
                              │    ┌────▼────────┐       │
                              │    │ Weaviate     │       │
                              │    │ Insert       │       │
                              │    └─────────────┘       │
                              └──────────────────────────┘
```

### Knowledge Base Integration

The `type` field on Knowledge already exists (`knowledge.py:47`). We can add a new type like `"external"` or `"pipeline"`:

- Create a Knowledge Base with `type="external"` for Octobox
- Each push creates File records linked to this KB via `KnowledgeFile`
- This allows the documents to show up in the KB UI, be queryable, and be assignable to chat models
- Access control works the same way (admins can share with specific groups)

### Authentication & Security

For government customers:

- **API Key auth** — already supported pattern in our external pipeline code (`X-API-Key` header)
- **Scoped keys** — tie API keys to specific knowledge bases/collections
- **Rate limiting** — FastAPI middleware, easy to add
- **Request validation** — Pydantic models for strict input validation
- **Audit logging** — log all ingestion events with source, user, timestamp

### Weaviate Considerations

- **Batch insert** — our `WeaviateClient.insert()` already uses `batch.fixed_size(batch_size=100)` for efficient bulk inserts
- **Collection per KB** — each knowledge base gets its own collection, which is the existing pattern
- **Schema flexibility** — Weaviate auto-schema handles additional metadata properties from external sources, though we pre-define core properties
- **Embedding consistency** — critical that all documents use the SAME embedding model. Our endpoint generates embeddings server-side, ensuring consistency

### Additional Discovery: Pre-Computed Embedding Support

`save_embeddings_to_vector_db()` at `retrieval.py:1276-1349` is a separate function that accepts chunks with pre-computed embeddings (fields: `text` + `embedding`). This bypasses parsing, chunking, AND embedding generation entirely. If Octobox uses the same embedding model, this provides a faster path — but embedding model consistency must be guaranteed.

### Additional Discovery: SyncProvider Abstraction

`backend/open_webui/services/sync/provider.py` defines `SyncProvider` (ABC) and `TokenManager` (ABC) interfaces. The factory at line 80 currently supports only `"onedrive"`, with a commented-out placeholder for `google_drive`. This abstraction layer could potentially be extended for an Octobox pull-based sync if needed, though the push API is simpler.

### Additional Discovery: Weaviate Multi-Tenancy

Weaviate supports multi-tenancy where each tenant gets a dedicated shard within a collection. This could be relevant for isolating Octobox data from other external sources while sharing the same collection schema. Each tenant gets physical data isolation — important for government deployments.

### Security Recommendations for Government API

For Octobox (serving government organizations), the practical minimum security stack:

- **API key authentication** in `Authorization: Bearer` header (pattern exists in `external_retrieval.py`)
- **TLS 1.3** for all data in transit
- **Rate limiting** via SlowAPI or fastapi-limiter with per-tenant limits
- **IP allowlisting** if Octobox has static egress IPs
- **Audit logging** of all ingestion requests (timestamp, tenant, document count, result)
- **Request size limits** (e.g., 10MB body, 500 chunks/request)
- **Strict Pydantic validation** on all input fields

For higher security: **mTLS** (mutual TLS) eliminates credential theft risk entirely, implementable at the load balancer level without application changes.

### Idempotency via Deterministic UUIDs

Use `uuid5(namespace, f"{document_id}:{chunk_id}")` to generate deterministic Weaviate UUIDs from Octobox's stable document/chunk IDs. This enables safe retries — re-sending the same document upserts rather than duplicates, complementing the existing hash-based dedup.

## Architecture Insights

1. **Embedding must happen server-side** — Weaviate is configured with `self_provided` vectors, meaning we MUST generate embeddings. External pipelines should NOT send their own embeddings (unless we add support for that separately), because all vectors in a collection must use the same embedding model.

2. **The `split=False` parameter is key** — `save_docs_to_vector_db()` already supports receiving pre-chunked data. Octobox just needs to send chunks, and we handle embedding + storage.

3. **File records are important** — The KB system tracks files via `File` + `KnowledgeFile` join tables. For external ingestion, we should still create File records (even without actual uploaded files) to maintain the UI's file listing and metadata display.

4. **Collection naming** — each KB uses a collection named after its ID. External documents should go into the KB's collection for proper RAG retrieval when the KB is attached to a model.

5. **The existing `process_text` endpoint is very close** — it already accepts text, creates a Document, and calls `save_docs_to_vector_db()`. A batch version with metadata support would be the minimal viable ingestion API.

## Open Questions

1. **Should we support pre-computed embeddings?** — If Octobox uses the same embedding model, we could accept their vectors directly and skip re-embedding. This would be faster but adds complexity and risks embedding model mismatch.

2. **Document lifecycle management** — When Octobox updates or deletes a source document, how should we handle it? Options: full replace by source ID, incremental updates, or manual cleanup.

3. **Rate limiting / quotas** — Should we impose per-tenant limits on ingestion volume? Important for multi-tenant government deployments.

4. **Knowledge base type** — Should the type be generic `"external"` or customer-specific `"octobox"`? Generic is more reusable but less specific for UI customization.

5. **Webhook for ingestion status** — Should we notify Octobox when ingestion completes/fails? Or is a synchronous response sufficient?

6. **Schema extension** — Octobox may have metadata fields not in our current Weaviate schema. Weaviate auto-schema handles this, but we should document which fields are indexed/searchable.
