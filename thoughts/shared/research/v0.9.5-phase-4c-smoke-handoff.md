## Open issues / questions for the next session

### 1. Agent still 401's against OWUI BE — even after `PROVIDER_OPENWEBUI_DIRECT_URL` fix

After making the agent reach the right port (18380), the user's "find similar research on the web" agent run errored with:

> Error code: 401 - {'error': 'Invalid username or password.'}

That message is what OWUI BE returns from `/api/v1/auths/signin` on bad credentials. The agent is calling some auth-protected OWUI endpoint with credentials that don't match. Likely candidates:

- `AGENTS_API_KEY` (agent side) vs `AGENT_API_KEY` (OWUI side) — must match
- Or the agent is hitting an endpoint that requires a user session bearer, not the inter-service API key

Bootstrap log confirmed the URL override took effect. So it's an auth-match issue, not a URL issue.

**Where to start:** grep `AGENT_API_KEY` and `AGENTS_API_KEY` in both repos, verify the .env.stack-3 contains both with matching values, and look at which OWUI endpoint returns "Invalid username or password" (that's not a generic message — it's specifically the signin endpoint).

### 2. Per-file Weaviate collections — efficiency regression vs. dev

The user noted (and I see in the logs):

```
generating embeddings for file-<uuid>
adding to collection file-<uuid>
added N items to collection file-<uuid>
```

Each uploaded file gets its own Weaviate collection (`file-<uuid>`). The user said **dev recently removed per-file collections** ("we do not create the per file KB anymore for efficiency"). Our merge branch is creating them again — Phase 1 / Phase 4d carve-out port restored the older flow.

Implications:
- Weaviate cluster bloat (lots of single-row collections).
- The "single-file selection from open KBs" UX the user described doesn't work anymore (since dev's design assumes files live only inside KB collections, not per-file).
- The KB-side embeddings still happen via base_worker's `_split_embed_and_store` (visible in earlier sync logs: `WEAVIATE KB INSERT START`) — those go into the KB collection directly. The per-file collections come from `process_file` → `save_docs_to_vector_db` with the default `collection_name = f'file-{file_id}'` (`routers/retrieval.py` around L1450).

**Where to start:** look at dev's `routers/retrieval.py:process_file` and `save_docs_to_vector_db` — what's the new flow? Probably the per-file collection step was removed and the file content goes straight into the KB collection. Compare against our carve-out and decide if we want to adopt that change as a follow-up commit on this branch or in a separate PR.

### 3. Concurrent file uploads still time out at the embedding step

Earlier this session (before the `e20b08b98` deadlock fix), uploading 5 PDFs in parallel resulted in 3 completing in ~5s and 2 timing out at `future.result(timeout=300)` 5 minutes later. After `e20b08b98` the **single-file** flow works cleanly. **The 5-concurrent case isn't re-tested** — might already be fixed (deadlock cascade), might still be there. Worth a smoke retry: upload 3–5 files at once and watch the logs.

If it still times out, look at:
- `RAG_EMBEDDING_CONCURRENT_REQUESTS=4` in primary .env — the semaphore is per-file, so 5 files × 4 = 20 concurrent requests to `api.inference.nebul.io/v1`. The embedding API may rate-limit, hold connections.
- anyio's default 40-thread thread-pool ceiling for `run_in_threadpool(save_docs_to_vector_db, ...)`.
- `httpx.AsyncClient` connection pool size.
