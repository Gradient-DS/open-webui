## Critical design questions for Phase 1.5 (decide before/during)

1. **`cleanup_worker.py` threading model.** Currently a sync background worker (likely runs in a thread started by `start_cleanup_worker()` in main.py lifespan). If services become async, the worker has two options:
   - **A.** Convert the worker itself to `async def` + run via `asyncio.create_task(...)` in lifespan (consistent with upstream's `scheduler_worker_loop`)
   - **B.** Keep the worker sync + use `asyncio.run(DeletionService.delete_user(...))` to bridge into async context per-call
   - Option A is cleaner; Option B is less invasive. Worth checking `start_cleanup_worker()` source first.

2. **ThreadPoolExecutor for vector cleanup.** `DeletionService` has a module-level `_vector_cleanup_pool = ThreadPoolExecutor(max_workers=10)` used for parallel `VECTOR_DB_CLIENT.delete_collection(...)` calls. Post-conversion options:
   - Keep the ThreadPoolExecutor + `loop.run_in_executor(_vector_cleanup_pool, _delete_collection, fid)` (preserves parallelism)
   - Replace with `ASYNC_VECTOR_DB_CLIENT` + `asyncio.gather(...)` (simpler, lets upstream's async client do its own concurrency)
   - The second is more aligned with upstream's direction but depends on whether `ASYNC_VECTOR_DB_CLIENT.delete_collection` is genuinely concurrent under the hood.

3. **`VECTOR_DB_CLIENT` vs `ASYNC_VECTOR_DB_CLIENT`.** DeletionService imports `from open_webui.retrieval.vector.factory import VECTOR_DB_CLIENT` (sync). If the service goes async, switching to `ASYNC_VECTOR_DB_CLIENT` is the natural step. But check that `ASYNC_VECTOR_DB_CLIENT` is a drop-in (same method names, same return types) before swapping.

4. **`DeletionService.delete_user` runs in a threadpool today.** The router (`routers/users.py:700`) wraps it via `await run_in_threadpool(DeletionService.delete_user, user_id)`. This was a deliberate workaround for the sync service blocking the event loop. Once the service is async, drop the wrapper and replace with `await DeletionService.delete_user(user_id)`.

5. **Sync model helpers staying sync.** Several of our Gradient-specific model methods (e.g., `Knowledges.update_knowledge_meta_by_id`, `Files.update_file_path_by_id`) are still sync — Phase 1.5 should convert these too, OR explicitly document why they stay sync (e.g., called only from already-threaded sync workers). Don't leave them as inconsistent half-async classes.
