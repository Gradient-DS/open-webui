## Plan corrections discovered during verification

### Correction 1 — child migration's `down_revision` must also be updated

The plan's Phase 0 step 5 says:

> Rename to: `d5e6f7a8b9ca_add_automation_tables.py`, `revision = "d5e6f7a8b9ca"`, **leave `down_revision` and content alone**.

That's true *for the renamed file itself* — but the child migration in upstream's chain points back at `d4e5f6a7b8c9` and **also** needs an update or it will resolve to OUR migration of that ID, fracturing the chain.

```
upstream/main:b7c8d9e0f1a2_add_last_read_at_to_chat.py
    down_revision = 'd4e5f6a7b8c9'   ← must become 'd5e6f7a8b9ca' after rename
```

**Action for Phase 1:** when the new migrations land in the merge, also edit `b7c8d9e0f1a2_add_last_read_at_to_chat.py` so its `down_revision` reads `'d5e6f7a8b9ca'`.

### Correction 3 — Phase 2's merge-node parents

Phase 2 step 3 says:

```python

```

This is **wrong** — `d5e6f7a8b9ca` is mid-chain, not the upstream top. With this parenting, alembic would still see two heads (the merge node AND `a0b1c2d3e4f5`).

**Correct merge-node parenting:**

```python

```

Where `f9a0b1c2d3e4` is **our** head (`add_2fa_grace_started`, verified against tenant DB `alembic_version`) and `a0b1c2d3e4f5` is **upstream's** top.
