## Learnings

1. **Prior handoff's open-issue #1 was misdiagnosed.** The "Invalid username or password" 401 error came from `https://router.huggingface.co/v1/chat/completions` (the underlying LLM provider) — NOT from OWUI's `/api/v1/auths/signin`. Easy to verify: grep `"Invalid username or password"` in `backend/open_webui/` returns nothing; the message lives upstream at the HF router. Stack trace shows `langchain_openai/chat_models/base.py:1846` (the LLM call inside `_call_model`), not OWUI. Lesson: when an agent flow 401s and the error string looks OWUI-shaped, **trust the stack trace**, don't pattern-match the message.

2. **Per-stack agent overlays must mirror `lex.yaml`'s default_llm fully when running against Nebul.** The stack-manager auto-creates an empty `stack-N.yaml` and that means deep-merge with `base.yaml` (which is HF-router-configured). Just overriding `base_url` (the pattern `feat/bim-agent` uses) leaves `model_id: openai/gpt-oss-120b:groq` inherited — which 404s on Nebul ("Model 'openai/gpt-oss-120b:groq' not found"). Override `model_id` AND `base_url` AND `api_key_env` AND `context_window_tokens` like `lex.yaml`.

3. **`utils/agent.py` is the lone Gradient call site that didn't get the Phase 1.5 async-conversion sweep.** All 11 other call sites of `get_event_emitter` in the tree (`functions.py`, `main.py`, `routers/chats.py`, `utils/actions.py`, `utils/chat.py`, `utils/middleware.py`) correctly `await` it. `utils/agent.py:422` was missed because it's pure Gradient code (the agent_proxy streaming path), which predates the upstream async conversion and isn't easily caught by upstream-diff scanning. **The pattern to look for in future merges:** `await x(...)` where `x` was assigned via `x = some_func(...)` rather than `x = await some_func(...)` — if `some_func` became async upstream, `x` is now a coroutine and the inner call will raise `'coroutine' object is not callable`. The error WILL be hidden if there's a `try/except Exception` around the call, as there was here.

4. **`event_type == 'panel_filter'` is real.** I'd only seen status/source/present_ui/context_usage referenced in `utils/agent.py` source, but the log also showed `Error emitting panel_filter event`. Worth double-checking that branch exists / is wired correctly when validating tomorrow — possibly another upstream-added event type that needs explicit handling.

5. **The visible "no status updates" symptom was specifically: model badge + answer + citations render fine, but no `StatusHistory` rows above the answer.** This is what the user described as "status updates got broken in this merge". The Socket.IO `events` room never got the `type: 'status'` payloads because the emitter call threw before reaching `sio.emit`. The frontend `chatEventHandler` at `src/lib/components/chat/Chat.svelte:538` is wired correctly; the bug was 100% backend.

6. **Earlier-screenshot confusion (for the record):** the user's first screenshot showed a floating "OI New Chat / Thought for 2 seconds" element near the top of the chat area. I burned a lot of investigation cycles assuming this was a misplaced StatusHistory render. It almost certainly was the **sidebar chat-item hover/focus tooltip** (browser-native or bits-ui Tooltip), and the actual complaint was the **absence** of inline status rendering, not its misplacement. **Lesson for next time:** when a user says "X is broken" + sends a screenshot, ask up-front whether the popup element IS X or whether they're pointing at the absence of X. Don't deep-dive a hypothesis without anchoring on a DOM inspection.
