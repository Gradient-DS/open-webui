# Phase 0.4 — AIOHTTP redirect-blocking spike

**Date:** 2026-05-24
**Source:** [`2026-05-24-open-webui-upstream-v0.9.5-merge.md`](../plans/2026-05-24-open-webui-upstream-v0.9.5-merge.md) Phase 0 step 4.

## Question

Will the new upstream `AIOHTTP_CLIENT_ALLOW_REDIRECTS` env var (default `False`) break any flow in our fork?

## Method

```bash
grep -rnE "allow_redirects" backend/open_webui/   # find ALL usages
grep -rnE "allow_redirects=True" backend/open_webui/   # find behavior-changing ones
git grep -nE "AIOHTTP_CLIENT_ALLOW_REDIRECTS" upstream/main -- '*.py'  # find where upstream wires it in
```

## Findings

### Our fork's `allow_redirects` call sites (5 total — all `False`)

| File:line | Value | Notes |
|---|---|---|
| `backend/open_webui/utils/code_interpreter.py:91` | `False` (literal) | Sandbox endpoint fetch — SSRF-conscious |
| `backend/open_webui/utils/tools.py:1283` | `False` (literal) | OpenAPI spec fetch |
| `backend/open_webui/utils/tools.py:1308` | `False` (literal) | OpenAPI spec fetch |
| `backend/open_webui/utils/oauth.py:635` | `False` (literal) | OAuth profile picture fetch (the SSRF-fix PR `435efa31c`) |
| `backend/open_webui/retrieval/web/utils.py:569` | `False` (literal) | Web search result fetch |

**Number of `allow_redirects=True` call sites in our backend: 0**

### How upstream wires `AIOHTTP_CLIENT_ALLOW_REDIRECTS`

The env var is **NOT** a session-wide monkey-patch. Upstream individually passes it as an argument at 9 call sites:

| Upstream file | Behavior |
|---|---|
| `retrieval/utils.py:189` | `requests.get(... allow_redirects=AIOHTTP_CLIENT_ALLOW_REDIRECTS)` |
| `retrieval/web/utils.py:505,524` | aiohttp session calls |
| `routers/images.py:820` | image fetch |
| `utils/code_interpreter.py:93` | (was hard-`False` in v0.8.12; now configurable) |
| `utils/files.py:69` | file URL fetch |
| `utils/oauth.py:744` | (was hard-`False`; now configurable) |
| `utils/tools.py:1437,1462` | (was hard-`False`; now configurable) |

**Other aiohttp call sites in upstream don't read this env var** — they default to aiohttp's built-in default (`True`).

## Conclusion

**No behavior change for our fork**. The env var only affects 9 specific call sites that upstream chose to make configurable, and all of them were already defaulting to no-redirect anyway. Our existing hard-coded `False` calls coincide with upstream's new env-var-bound calls.

### Optional Phase 1 follow-up

When resolving merge conflicts in these four files, **adopt the env-var-bound pattern** to give admins the escape hatch upstream intended:

```python
# instead of: allow_redirects=False  (current ours, hardcoded)
# use:        allow_redirects=AIOHTTP_CLIENT_ALLOW_REDIRECTS  (after merge)
```

Affected files:
- `backend/open_webui/utils/code_interpreter.py:91`
- `backend/open_webui/utils/tools.py:1283,1308`
- `backend/open_webui/utils/oauth.py:635`
- `backend/open_webui/retrieval/web/utils.py:569`

Default behavior stays identical (`False`), only the configurability changes. **Not blocking the merge**; flag as a nice-to-have during Phase 1 conflict resolution.

### Out-of-scope reminders

- **OneDrive / Google Drive / Confluence cloud sync** — `httpx`-based, completely unaffected by this env var. No changes.
- **Frontend file pickers** (`onedrive-file-picker.ts`, `google-drive-picker.ts`) — browser `fetch()`, unaffected.
- **LLM proxy aiohttp calls** (Ollama, OpenAI, agent proxy) — POST-to-API, never expect 3xx, would default to `True` anyway. No changes.
